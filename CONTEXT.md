# Pi Telegram Bot

This context describes the language for a Telegram-based gateway that lets an authorized user control Pi Coding Agent sessions remotely.

## Language

**Bot Gateway**:
A Telegram-facing service that accepts authorized Telegram messages and translates them into Pi RPC commands.
_Avoid_: Telegram app, bridge, wrapper

**Pi Session**:
A persistent Pi conversation stored as a `.jsonl` session file under the configured Pi session directory.
_Avoid_: chat, thread, log

**Pi Session Directory**:
The directory where the Bot Gateway discovers and stores Pi Session files.
_Avoid_: workspace, project directory, session store

**Authorized Telegram User**:
A Telegram `from.id` that is allowed to control the Bot Gateway.
_Avoid_: admin, account, member

**Telegram Chat**:
The Telegram destination identified by `chat.id` where the Bot Gateway sends replies and remembers transient command context.
_Avoid_: user, account

**Session List**:
An in-memory, per-Telegram Chat numbered view of Pi Sessions generated by `/sessions` using session file stems as display names.
_Avoid_: registry, database, persisted index

**Pi Session Name**:
The optional display name for the active Pi Session managed through Pi RPC.
_Avoid_: filename, title

**Follow-up Prompt**:
A normal Telegram text message queued for Pi to process after the current streaming run finishes.
_Avoid_: steer, interrupt, background task

**Buffered Answer**:
The assistant text accumulated from Pi streaming events and sent to Telegram after Pi emits `agent_end`.
_Avoid_: live stream, tool log, transcript

**Safe Error**:
A short Telegram-facing failure message that avoids raw stderr, stack traces, secrets, and unnecessary local file details.
_Avoid_: debug dump, stack trace, raw error

## Relationships

- A **Bot Gateway** accepts messages from one or more **Authorized Telegram Users**.
- A **Bot Gateway** sends replies to the **Telegram Chat** associated with each incoming message.
- Authorization is based on **Authorized Telegram User**, not **Telegram Chat**.
- A **Bot Gateway** controls exactly one active **Pi Session** at a time in the MVP, shared by all **Authorized Telegram Users**.
- A **Pi Session** is selected from `.jsonl` files directly inside the **Pi Session Directory** or created fresh by the **Bot Gateway**.
- The **Pi Session Directory** comes from `PI_SESSION_DIR`, defaulting to `~/.pi/sessions` when unset.
- A **Session List** belongs to exactly one **Telegram Chat** and is regenerated by `/sessions`.
- `/use <id>` selects from the latest **Session List** for that **Telegram Chat**.
- A **Session List** displays each **Pi Session** by file stem, not by parsing the session file.
- `/current` displays the **Pi Session Name** when Pi RPC provides one, otherwise the active session file stem.
- A **Follow-up Prompt** is created when an **Authorized Telegram User** sends normal text while Pi is already streaming.
- A **Buffered Answer** is sent after Pi completes a run; MVP does not expose live streaming text or tool events by default.
- Each **Buffered Answer** is sent to the **Telegram Chat** that submitted the prompt that produced it.
- A **Safe Error** is sent when Pi RPC is unavailable or a Pi RPC command fails.

## Example dialogue

> **Dev:** "When an **Authorized Telegram User** sends normal text, should the **Bot Gateway** create a new **Pi Session** every time?"
> **Domain expert:** "No — normal text is sent to the currently active **Pi Session** unless the user explicitly runs `/new` or `/use <id>`."

## Flagged ambiguities

- "session" means **Pi Session**, not a Telegram chat session.
- "user" means **Authorized Telegram User** when discussing access control; **Telegram Chat** is only the reply destination.
